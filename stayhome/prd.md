# [PRD] 사내 제휴 리조트 통합 조회 시스템

## 프로젝트명: Welfare Stay

---

# 1. 프로젝트 개요 (Overview)

## 1.1 배경

현재 복지 담당자는 사내 직원들의 제휴 리조트 예약 문의를 처리하기 위해 제휴 리조트 사이트에 각각 로그인하여 잔여 객실을 직접 확인하고 있다.

이 과정에서 아래와 같은 문제가 발생한다.

* 동일한 조회 작업 반복
* 리조트별 로그인/검색 절차 상이
* 응답 시간 증가
* 업무 피로도 증가
* 특정 사이트 장애 시 전체 조회 흐름 중단

특히 단순 “조회 업무”에 과도한 시간이 사용되고 있으며, 반복적인 브라우징 작업이 핵심 비효율로 작용하고 있다.

---

## 1.2 목적

6개 제휴 리조트의 잔여 객실 정보를 단일 화면에서 조회할 수 있는 개인용 통합 조회 시스템을 구축한다.

서비스는:

* 빠른 검색 속도
* 최소 클릭
* 모바일 사용성
* 운영 부담 최소화

를 목표로 한다.

---

## 1.3 서비스 성격

본 서비스는:

* 사내 공개 서비스가 아닌
* 복지 담당자 개인 1인이 사용하는 내부 도구이며
* 운영 편의성과 유지보수성을 우선한다.

---

## 1.4 KPI

| 항목           | 기존     | 목표     |
| ------------ | ------ | ------ |
| 리조트 조회 시간    | 10~15분 | 30초 이내 |
| 반복 조회 작업     | 수동     | 자동화    |
| 리조트 확인 페이지 수 | 최대 6개  | 1개     |
| 모바일 접근성      | 없음     | PWA 지원 |

# 2. 사용자 시나리오 (User Scenario)

## 2.1 주요 사용자

* 복지 담당자 1인

---

## 2.2 핵심 시나리오

### Scenario 1

복지 담당자가 직원 문의를 받는다.

```text id="1k4qxa"
"이번 주말 제주 지역 예약 가능한 리조트 있나요?"
```

담당자는:

1. Welfare Stay 접속
2. 날짜 + 지역 선택
3. 조회 클릭
4. 통합 결과 확인

을 통해 즉시 응답 가능해야 한다.

---

### Scenario 2

특정 리조트 크롤링 실패 발생

시스템은:

* 전체 조회를 중단하지 않고
* 실패한 리조트만 오류 표시
* 나머지 결과는 정상 제공

해야 한다.

# 7. 시스템 아키텍처

## 7.1 기술 스택

| 영역         | 기술                      |
| ---------- | ----------------------- |
| Frontend   | Next.js App Router      |
| UI         | shadcn/ui + TailwindCSS |
| Mobile     | PWA                     |
| Hosting    | Vercel                  |
| Database   | Neon PostgreSQL         |
| ORM        | Prisma                  |
| Crawling   | Playwright              |
| Auth       | NextAuth                |
| Scheduler  | Vercel Cron             |
| State      | React Query             |
| Logging    | Neon DB                 |
| Monitoring | Slack Webhook           |

---

## 7.2 아키텍처 흐름

```text id="zg3ryj"
사용자 조회
    ↓
Next.js API Route
    ↓
Neon DB 캐시 조회
    ↓
데이터 반환

(백그라운드)

Vercel Cron
    ↓
Playwright 크롤링
    ↓
Neon DB 저장
```
